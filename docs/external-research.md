## A. 兩個網站功能盤點

### A.1 L22203 — 數據小熊熊歷險記

- **主題**:資料處理工具與特徵工程,涵蓋:
  - Robust Scaling(穩健標準化對抗極端值)
  - 高基數特徵編碼陷阱
  - SMOTE 不平衡資料
  - Apache Flink 即時串流、Spark 記憶體運算
  - 過濾法(Filter Method)特徵選擇
  - Label vs One-hot 編碼選擇
  - 交互特徵(Interaction Feature)
  - 目標編碼資料洩漏風險
  - Apache Airflow 管線排程
  - 共 10 道單選題

- **互動方式**:
  - **三大功能分支**:首頁 → 圖解教室(10 張卡)/ 讀教材(6 堂課)/ 實戰測驗(隨機 10 題)
  - 圖解教室:卡片自動播放 + 上/下一張按鈕
  - 教材庫:6 堂課依序學習(類別編碼大補帖、特徵工程魔法、分散式處理四種框架...)
  - 測驗:選 ABCD → 即時顯示「正確/錯誤」+ 詳細解析 + 角色表情變化
  - 答對 +10 分 + 蜂蜜表情;答錯 -1 HP + 受傷表情
  - HP 歸零遊戲結束;結算依 HP/分數三檔評價(👑 大師 / 🎉 通關 / 🩹 鼓勵重來)

- **UI 風格**:
  - 卡通溫暖風,Tailwind 漸層卡片 + Emoji 動畫
  - 主角是熊熊,有 6 種表情狀態:🐻 正常 / 🤓 學習 / 🍯 答對 / 🤕 答錯 / 😵 失敗 / 🧸 通關
  - 響應式(`md:` 斷點),手機桌面通吃

- **鐵律相容性**:
  - 鐵律 #1(下鑽):答錯只給文字解析,**沒有結構化變化型下鑽** → 弱於本案
  - 鐵律 #2(動態化):題目隨機打亂順序 ✓,但 ABCD 不洗、無變數池
  - 鐵律 #3(原創):看似自寫,無法核實是否抄 114-2(本案無風險)
  - 鐵律 #4(選項長度):無法從外部驗證
  - 鐵律 #5(來源忠實):**主題核心是 L22203 純科二 Hadoop/Spark/Flink/Airflow,本案 scope 已設 include:false** → 若我們直接複製此主題會違反鐵律 #5

- **學習助益**:
  - 🟢 **三段式學習迴圈值得借鏡**:先看圖鑑(視覺記憶)→ 讀教材(系統理解)→ 做測驗(檢核)。對「臨考前才開始」的考生友善
  - 🟢 角色表情反饋強化情感連結,降低答錯壓力
  - 🟢 圖解教室的「動畫 + 文字」雙模態幫助記憶抽象概念(SMOTE、Robust Scaling 公式)
  - 🔴 缺少結構化下鑽,單純答對答錯無法深化

### A.2 L22103 — 小獅子的統計大冒險

- **主題**:假設檢定與統計推論,24 關涵蓋:
  - 母體參數 vs 樣本統計量
  - z 檢定 / t 檢定 / ANOVA / 卡方檢定 / 非參數檢定
  - 型一/型二錯誤、顯著水準 α、p-value
  - 相關係數(Pearson、Spearman)
  - 因果謬誤(相關 ≠ 因果禁忌)

- **互動方式**:
  - **單線闖關**:24 關按順序解鎖,進度條點點(綠=已通關、橙=當前、灰=未開啟)
  - 每關一題三/四選一 → 答題即時反饋
  - **3D 場景**:Three.js 建構統計森林(150 棵樹、400 朵花、10 朵雲、小獅子主角)
  - **相機操作**:左鍵拖曳旋轉、滾輪縮放(5~30 單位)、觸控相容
  - 小獅子有狀態機(idle / walk / jump / shake)
  - 答對:綠色 overlay + 小獅子跳躍 + 進度點轉綠
  - 答錯:紅色訊息 + 小獅子搖頭 + 重試
  - **失誤計數**影響結局評價(零失誤 / 優秀 / 努力)
  - 通關時小獅子戴皇冠

- **UI 風格**:
  - 卡通 3D,森林暖色系
  - 全畫面沉浸,操作說明用 emoji(🖱️ 🔍 🎉)
  - 鼓勵性文案

- **鐵律相容性**:
  - 鐵律 #1(下鑽):**只有重試**,沒有變化型下鑽 → 弱於本案
  - 鐵律 #2(動態化):24 關固定順序,**沒有洗牌也沒有變數池** → 重玩死記
  - 鐵律 #3(原創):無法核實
  - 鐵律 #4(選項長度):無法從外部驗證(很多三選一)
  - 鐵律 #5(來源忠實):**L22103 = boundary_overlap,本案 scope 接受** → 主題本身沒問題,但 24 關內容若涉及純科二細節(如 Levene 變異數同質檢定的 SPSS 操作)需要驗證

- **學習助益**:
  - 🟢 **3D 沉浸 + 場景式記憶錨點**:把抽象的「型一錯誤」綁到「小獅子被陷阱絆倒」的視覺,記憶效果強
  - 🟢 **單線闖關 + 進度點**符合「破關癮」遊戲化心理,適合每天 10 分鐘衝一關
  - 🟢 失誤計數 → 結局評價,有「再戰一次拿零失誤」的二次遊玩動機
  - 🔴 沒有自適應(每位玩家走相同 24 關)
  - 🔴 3D 場景開發成本高,移動端效能與電量消耗大

## C. 提案的案 6 + 案 7

### 提案 1(主推)— 案 6:**「知識卡牌圖鑑 — 收集式認知預習」**

- **名稱**:`Mode 6 · 卡牌圖鑑大全`(代號:Codex 模式)
- **靈感來源**:L22203 的「圖鑑教室」+ Pokémon Go 的圖鑑收集機制
- **玩法**:
  1. 進入後是一個「**87 張(對應 87 個 kb 真實節點)知識卡片大圖鑑**」,初始全部灰色
  2. 點任一卡片 → 進入「卡片詳情頁」,顯示該節點的 kb summary / key_points / misconceptions / hook
  3. 詳情頁底部「**封印挑戰**」按鈕 → 從該 node_id 抽 2~3 題(從 325 題既有題庫)
  4. **答對 1 題**:卡片從灰 → 半透;**答對 2 題**:卡片解鎖(彩色 + 閃光);**全對**:卡片變「金卡」並寫入 `Mastery.drillBonus()` +20
  5. 圖鑑頁顯示 **進度條:已解鎖 / 已金卡 / 全部**,並對未解鎖卡片畫面 **rotation 漸層動畫**(該你來收集!)
  6. 解鎖卡片可在「**對戰倉**」拼成「卡組」(選 5 張) → 進入「卡組挑戰」連戰 5 題,全對解鎖「卡組徽章」(進階成就)

- **知識點 / 題型基礎**:
  - 不需新增題庫:**直接讀現有 325 題的 node_id 集合**,做一個 `nodes-with-questions` 索引
  - 各 format(single_choice/calculation/code_reading/matching/sequence)都可用,因為單張卡可以隨機從該 node 的所有題型抽
  - kb 文字內容直接從 `kb/nodes-subject-1.json` / `nodes-subject-3.json` 讀

- **為何值得加(學習價值)**:
  - 🟢 **填補本案 5 案的「教學段」缺口**:目前考生只能「做題 → 看解析」,沒有「主動翻書認識知識點」的入口。卡牌圖鑑提供「**認知預習**」(看 kb 內容)→「**檢核**」(封印挑戰)兩段式,符合教育學的 expository → reinforcement loop
  - 🟢 **強化長期動機**:鐵律 #5 限定 87 個節點,87 張卡有「**全收集慾望**」(Pokémon 效應)。考前 13 天每天解鎖 7 張剛好破關
  - 🟢 **與既有系統零衝突**:不改題庫、不改共用層、純讀 kb + QUESTIONS,完全不會影響鐵律
  - 🟢 **天然支援鐵律 #1 下鑽**:每張卡的「封印挑戰」答錯時可直接接 `DrillSession.start()`,完美對接

- **預估開發複雜度**:**中-低**
  - mode6.js 約 600~800 行(類比 mode4.js)
  - 主要工作:卡片網格 UI + 卡片翻轉動畫(GSAP)+ 圖鑑進度持久化(`Storage.set('ipas_mode6_codex_v1', ...)`)+ 卡組挑戰連戰邏輯
  - 不需 3D、不需新題型、不需新 schema

- **鐵律 #5 限制**:
  - **完全合規**:只用既有 87 個 kb 節點 + 既有 325 題
  - 不需新增題型,不需新增 kb 節點

### 提案 2(次推)— 案 7:**「考古題情境劇場 — 30 分鐘模考連續劇」**

- **名稱**:`Mode 7 · 考古題模考劇場`(代號:Theater 模式)
- **靈感來源**:L22103 的「24 關闖關」+ 真實考試「30 分鐘 25 題」的時間壓力
- **玩法**:
  1. 進入後選擇「**模考劇本**」:科一 30 題 / 科三 25 題 / 混合衝刺 50 題
  2. 進入後是「**情境劇場**」:畫面分為兩半,左邊是「**情境角色 + 對話框**」(類似 mode1 的 RPG 場景),右邊是題目卡
  3. 角色配置(共 5 位 NPC,各代表一條學習主線):
     - 👨‍💻 工程師「阿凱」:程式判讀題出場
     - 👩‍🔬 數據科學家「莉雅」:特徵工程 / 統計題出場
     - 🤖 模型「Trans 哥」:深度學習 / 生成式題出場
     - 👮 倫理委員「方爺」:風險 / 治理 / 公平性題出場
     - 👨‍💼 顧問「林姊」:導入規劃 / 情境決策題出場
  4. 每答完一題,角色根據答對/答錯說一句話(口語化,具角色個性)
  5. **計時器**:30 分鐘倒數(模擬真實考試),時間結束強制交卷
  6. **進度條**:25 顆星按關卡推進,答對轉金 / 答錯轉紅
  7. 結算:**Pace Heatmap**(每題用時)+ **錯題回顧**(可一鍵丟進錯題本下鑽)+ **科目得分**(分科一 / 科三)+ **預估等級**(對標 IPAS 中級門檻 60 分)

- **知識點 / 題型基礎**:
  - 不需新增題庫:**從 325 題抽樣**,依任務類型分配:
    - 科一模考:knowledge_code 開頭 L21*
    - 科三模考:knowledge_code 開頭 L23*
    - 混合衝刺:依現實考試比例(科一 40%、科三 50%、L22 邊界 10%)
  - 任意 format 都可,題庫已夠

- **為何值得加(學習價值)**:
  - 🟢 **填補「臨場壓力訓練」缺口**:現有 5 案都允許「停下來想很久」,但真考試 IPAS 中級科一是 60 分鐘 50 題(平均 72 秒/題)。沒有計時模考會讓考生第一次面對時間壓力時崩潰
  - 🟢 **多角色情境讓答題不無聊**:1 場 25 題若沒情境包裝,大腦會疲乏。透過 5 位 NPC 輪流出場(每位 5 題)製造「這題接下來換誰?」的小期待
  - 🟢 **Pace Heatmap 是真有用的學習工具**:讓考生看到自己「卡題」的位置(連續 3 題用時 > 2 分鐘 = 該知識區弱),配合錯題本下鑽
  - 🟢 **預估等級**給考前 13 天的「自評天花板」:60 分?70 分?80 分?讓考生知道還差多少

- **預估開發複雜度**:**中**
  - mode7.js 約 800~1000 行
  - 主要工作:30 分鐘 timer + 角色對話系統(可重用 mode1 的對話框 CSS)+ Pace Heatmap(SVG 條狀圖)+ 結算頁(分科目得分計算)
  - 重用率高:`PlayEngine.show` / `Wrongbook.add` / `Mastery.update` 全部現成
  - 不需 3D、不需新題型

- **鐵律 #5 限制**:
  - **完全合規**:只從 325 題抽,所有 knowledge_code 都已合法
  - 不需新增題型,不需新增 kb 節點
  - 角色對話文案需注意:**口語化但不洩漏答案**(避免「不對啦,要選 B!」這種破壞題目效用)
